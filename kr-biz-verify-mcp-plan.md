# 작업계획서 — 한국 비즈니스 검증 액션 MCP 서버
**버전 v1.0 (2026-07-07) · 클로드 코드 루프 엔지니어링용**
**작업명(가칭): `kr-biz-verify` (최종 명칭은 Loop 1에서 확정)**
**전제: A안(`kr-disclosure-intel`) 계획서 v1.2의 자매 문서. A안 자산 재사용을 전제로 한 경량판.**

---

## 0. 개요와 전략

### 0.1 제품 한 줄 정의
한국 사업자·거래처에 대한 **검증 행동**(진위확인, 휴폐업, 과세유형, 세금계산서 발행 가능 여부)을 AI 에이전트에게 도구로 제공하는 MCP 서버.

### 0.2 A안과의 관계 — 같은 논리, 다른 시장
| | A안 (kr-disclosure-intel) | B안 (kr-biz-verify) |
|---|---|---|
| 성격 | 정보 인텔리전스 (읽기) | 검증 액션 (판정) |
| 데이터 소스 | DART (일 4만 콜) | 국세청 진위확인·상태조회 API (일 100만 콜, 1회 100건 배치) |
| 주 타깃 | 영어권 투자자·에이전트 개발자 | **한국 기업·한국 개발자** (경비처리·거래처 온보딩·정산 자동화) |
| 부 타깃 | 소형 해외 기관 | 한국 진출 해외 이커머스·핀테크 |
| 문서 언어 | 영어 우선 | **한국어 우선 + 영어 병기** |
| 해자 | 한국어 파싱 + 영어 출력 | **제품화 선점** (수요는 검증됐으나 MCP 제품 부재) |
| 기술 난도 | 높음 (XBRL·비정형 파싱) | 낮음 (정형 API 래핑 + 판정 로직) |
| 경쟁 | koreanpulse 등 경쟁 시작 | 취미 프로젝트·범용 래퍼뿐, 제품화 사례 없음 |

**함의:** B안은 파싱 해자가 없는 대신 실행 속도가 해자다. 기술 난도가 낮으므로 **빨리 만들고, 빨리 등재하고, 신뢰성(가동률·에러 처리)으로 방어**한다.

### 0.3 핵심 판단 근거 (요약)
- 국세청 공식 API의 존재 이유 자체가 실무 통증(플랫폼의 공급사업자 검증, 법인카드 대량 확인, 간이과세자 세금계산서 의무화에 따른 과세유형 확인) — 국가가 수요를 인정한 영역.
- 그 실무가 AI 에이전트로 이관되는 중인데 에이전트용 어댑터(MCP)가 없음.
- 검증은 성공/실패가 명확 → **건당 과금**의 이상형.
- 일 100만 콜 한도 → A안의 쿼터 걱정이 구조적으로 부재.

### 0.4 A안 자산 재사용 지도
| A안 산출물 | B안에서의 용도 | 수정 정도 |
|---|---|---|
| Loop 0 스캐폴딩 (TS+SDK+CI 구성) | 저장소 복제 후 이름만 교체 | 5% |
| Loop 2 `quota.ts`·`cache.ts` | 쿼터 로직 단순화(한도 여유), 캐시는 상태조회 TTL용 | 30% |
| Loop 5 MCP 서버 골격·도구 등록 패턴 | 그대로 | 10% |
| Loop 7 README 템플릿·DEMO_MODE 패턴 | 언어 우선순위만 반전(한국어 우선) | 20% |
| Loop 8 디렉터리 제출 파이프라인·문안 | 대상 동일, 문안 교체 | 20% |
| CLAUDE.md 불변 원칙·게이트 | 도메인 규칙만 교체 | 15% |
| Loop 10 MoR 결제 결정(Polar/Paddle) | 동일 적용. 단 주 타깃이 한국 고객이므로 **국내 PG(토스페이먼츠 등) 병행 검토** 추가 | 신규 30% |

**착수 시점 원칙(재확인):** A안 Loop 8 완료 후 착수가 기본. 예외 ① A안이 Loop 1에서 기각된 경우 → B안 즉시 단독 착수(이때 위 표의 "재사용"은 B안에서 신규 구축로 전환), ② A안 대기 시간에 할 수 있는 것은 §6 체크리스트의 키 신청뿐.

## 1. 도구 설계 (제품의 핵심)

### 1.1 도구 5종
1. **`verify_business(business_no, representative_name, opening_date)`**
   국세청 등록정보와의 일치 여부 판정(진위확인). 반환: `match: true/false` + 불일치 시 어느 필드가 다른지는 국세청이 알려주지 않음을 명시.
2. **`check_business_status(business_no)`**
   휴업·폐업 여부, 과세유형(일반/간이/면세), 폐업일자. 가장 호출 빈도 높을 도구.
3. **`batch_check_status(business_no[])`**
   최대 100건 일괄 상태조회(국세청 API의 1회 100건 배치와 1:1 대응). 경비처리·정산 에이전트의 핵심 도구.
4. **`check_invoice_eligibility(business_no)`**
   상태조회 결과에 **판정 로직**을 얹은 부가가치 도구: 과세유형·휴폐업 상태를 종합해 "세금계산서 수취 가능 / 계산서 대상(면세) / 발행 불가(폐업)" 판정 + 근거. **순수 래퍼와의 차별화 지점.**
5. **`explain_kr_tax_type(tax_type)`**
   일반과세·간이과세·면세의 실무 의미 해설(정적 지식, 무료 미끼 — A안 `explain_event_type`과 동일 역할).

### 1.2 설계 원칙
- 반환은 zod 스키마 고정. 판정(`eligibility`)에는 반드시 `basis`(근거 필드)를 동봉 — 에이전트가 사용자에게 근거를 제시할 수 있게.
- **개인정보 원칙:** 대표자명 등 입력값은 검증 요청 즉시 폐기, 로그·캐시에 저장 금지(사업자번호와 상태 결과만 캐시 가능). CLAUDE.md 불변 원칙에 포함.
- 캐시 TTL: 상태조회 24시간(휴폐업은 급변하지 않음), 진위확인은 **캐시 금지**(입력 조합별 판정이라 캐시 의미 없고 개인정보 원칙과 충돌).
- 면책: 출력에 "국세청 공공데이터 기준, 법적 효력 있는 증명은 홈택스 발급 문서 참조" 상시 포함.

## 2. 루프 정의 (Loop 0 ~ Loop 8)

A안보다 루프가 적다(파싱·번역 계층이 없으므로). 각 루프 구성: 목표 / 작업 / AC / 재시도.

### Loop 0 — 스캐폴딩 (A안 복제)
- **작업:** A안 저장소를 템플릿으로 복제, 명칭·패키지명 교체, CI 통과 확인. CLAUDE.md의 도메인 규칙 교체(§1.2 개인정보 원칙 반영).
- **AC:** 빌드·테스트·린트 통과, Inspector 연결(도구 0개).

### Loop 1 — 수요 검증 + 명칭 확정 `[HUMAN]`
- **목표:** "누가 이걸 매주 쓰는가"의 구체화. A안 Loop 1보다 경량이나 생략 불가.
- **작업:**
  1. 사용 시나리오 5종 작성(경비처리 에이전트, 거래처 온보딩, 이커머스 셀러 검증, 세무 자동화, 조달 실사) — 각각 "어떤 에이전트가, 어떤 트리거로, 어떤 도구를, 얼마나 자주" 호출하는지 서술.
  2. 국내 개발자 커뮤니티(GitHub 한국 공공데이터 관련 저장소, 관련 블로그·커뮤니티)에서 사업자 검증 자동화 시도 사례 수집 → 시나리오별 실수요 매핑.
  3. 기존 유료 대체재 가격 조사(사업자정보 API 상용 서비스들의 건당 단가) → 우리 가격의 준거점.
  4. 산출물: `docs/adr/ADR-001-demand.md`.
- **AC:** 5개 시나리오 중 3개 이상에서 실수요 증거(실제 시도 사례 또는 유료 대체재 존재) 확보.
- **[HUMAN]:** 명칭 확정(한국어 검색어 `사업자`, `진위확인` + 영어 `korea`, `business`, `verify` 포함 검토), npm 선점.
- **재시도:** 3개 미달 시 시나리오 재발굴 1회. 재실패 시 보류 보고(A안에 집중).

### Loop 2 — 국세청 API 클라이언트
- **작업:** 공공데이터포털 인증(서비스키), 진위확인·상태조회 두 엔드포인트 클라이언트, 배치(100건) 지원, 재시도·타임아웃, 공공 API 점검시간·일시 장애 대응(명시적 에러 코드), 상태조회 24h 캐시.
- **AC:** 실 API 통합 테스트(본인 사업자번호 등으로) 통과. 장애 주입 시 우아한 실패. 대표자명이 로그·캐시 어디에도 남지 않음을 grep 검증.
- **재시도:** 응답 포맷 이슈 시 파서 분리.
- **[HUMAN] 선행:** 공공데이터포털 가입 + 국세청 API 활용신청 승인(§6 체크리스트 — **지금 신청 가능**).

### Loop 3 — 도구 5종 구현 + 판정 로직
- **작업:** §1.1의 5개 도구. `check_invoice_eligibility`의 판정 규칙표(과세유형×상태 → 판정×근거)를 `docs/rules.md`로 먼저 문서화한 뒤 구현(규칙이 곧 스펙).
- **AC:** Inspector에서 5종 호출 성공. 판정 규칙표의 전 조합에 대한 테이블 주도 테스트 통과. Claude Desktop 실연결에서 "이 사업자번호 폐업했는지 확인해줘" 자연어 질의가 올바른 도구 호출로 이어짐.
- **재시도:** 도구 오선택 시 description 퇴고(A안 Loop 5와 동일 원칙).

### Loop 4 — 신뢰성 강화
- **작업:** 커버리지 80%+, 에러 코드 표준화, 잘못된 사업자번호 형식의 사전 검증(체크섬 — 국세청 호출 전에 형식 오류를 걸러 쿼터·지연 절약), `npm audit` 클린.
- **AC:** 커버리지 80%, 카오스 테스트(API 5xx·타임아웃 주입) 크래시 0.

### Loop 5 — 문서화 + DEMO_MODE (한국어 우선)
- **작업:** A안 템플릿 재사용하되 반전 — README.md는 한국어, README-EN.md 병기. DEMO_MODE: 가상 사업자번호 셋(실번호 사용 금지 — 개인정보·오인 방지)으로 키 없이 5종 도구 체험. 공공데이터포털 활용신청 절차 스크린샷 가이드(`docs/get-api-key.md`).
- **AC:** 키 없이 3분 내 첫 호출(DEMO). 라이브 전환 20분 내.

### Loop 6 — 디렉터리 총등재 `[HUMAN]`
- **작업:** A안 Loop 8 파이프라인 재사용. 동일 5+ 디렉터리 + 한국 개발자 채널(국내 커뮤니티·awesome-mcp-korea) 가중.
- **AC:** npm 게시 + 5개 이상 등재.

### Loop 7 — 호스티드 (게이트: 출시 4주 후 WAU 10)
- **작업:** A안 Loop 9와 동일 패턴. 단 검증은 실시간성이 생명이라 캐시 전략이 단순(상태조회만 24h). 배치 대량 처리용 비동기 잡 큐 추가 검토.
- **AC:** 호스티드 원클릭 연결 + 배치 100건 E2E.

### Loop 8 — 수익화 (게이트: 호스티드 WAU 30)
- **작업:** 건당 과금 우선(검증 1건 = 명확한 단위). 무료: 일 N건 + DEMO / 유료: 건당 또는 월정액 무제한. 결제는 A안 결정 재사용(MoR: Polar/Paddle) + **주 타깃이 한국 고객이므로 국내 PG 병행 검토**(토스페이먼츠 등 — 한국 카드·계좌이체 수용성).
- **[HUMAN]:** 가격(Loop 1의 유료 대체재 단가를 준거로), 결제 수단 조합, 사업자·세무.
